Запиши итоги текущей сессии в журнал `Resources/Session Log/`.

## Инструкция

1. Выполни две команды параллельно:
   - `date '+%Y-%m-%d %H:%M %z'` — точное время с часовым поясом
   - `bash scripts/session-analyze.sh` — метрики сессии
2. Определи ISO-неделю из даты. Файл журнала: `Resources/Session Log/YYYY-WNN.md` (например `2026-W12.md`). Неделя: понедельник–воскресенье (ISO 8601). Если файла нет — создай:
   ```
   # YYYY-WNN – Журнал сессий

   > Неделя: YYYY-MM-DD (Пн) — YYYY-MM-DD (Вс)
   ```
3. **Краевой случай (смена недели):** если `session_start` из анализатора попадает в предыдущую неделю — сессия началась на прошлой неделе. Запись пишется в файл **текущей** недели с пометкой `(начало в W{N-1})`.
4. Проанализируй, что было сделано в этой сессии. Для каждого пункта определи:
   - **Что** изменилось (действие, артефакт)
   - **Зачем** (бизнес-цель, проблема)
   - **В каком контексте** (проект, область жизни, домен)
5. **Проверь: есть ли уже запись с тем же `session:` ID в журнале** (текущая + предыдущая неделя). Если session-analyze вернул session_id — ищи строку `session: <id>`.
   - **Есть совпадение** → обнови существующую запись
   - **Нет совпадения** → добавь запись **в начало файла** (после заголовка и `>`), сохраняя обратную хронологию

## Формат записи

```markdown
### YYYY-MM-DD HH:MM

**Сессия:** начало HH:MM, промптов N, активное время ~Xч Yмин, пауз: K | параллельных сессий: M | локация
**Контекст:** [домен/область — например: Компания/продукт, LifeOS/инфраструктура, Личное/финансы]
<!-- session: SESSION_ID -->

- [Контекст] Что сделано и зачем. Если связано с проектом — wikilink.
- [Контекст] Второй пункт...

Изменено: `файл1`, `файл2`
```

## Контекст записи (главное правило)

Журнал сессий — **единственный источник данных** для агентов верхнего уровня (обзор недели, персональный коуч). Эти агенты не имеют доступа к transcript сессии. Они видят только записи в журнале.

Каждая запись должна содержать достаточно контекста, чтобы внешний агент мог:
- Понять, в каком домене/проекте/области жизни велась работа
- Определить, были ли параллельные сессии в одном контексте (нормально) или в разных (переключение фокуса)
- Оценить продуктивность и паттерны работы
- Учесть локацию и часовой пояс

**Строка «Контекст:»** — классификация домена. Не техническая («писал bash-скрипт»), а смысловая («LifeOS/инфраструктура — трекинг сессий для анализа продуктивности»).

**Пункты работы** — с контекстом в скобках, если затронуто больше одного домена. Один домен — скобки не нужны.

**Локация** — из анализатора (`location=`). Если `tz_changed=true` — часовой пояс менялся (перелёт), отметить.

## Правила

- Время в заголовке — точное (из `date`), момент вызова /session-log
- Строка **Сессия:** — метрики из session-analyze.sh. Если анализатор недоступен — опустить, не блокировать запись
- HTML-комментарий `<!-- session: ID -->` — технический маркер для обновления. Не удалять
- Каждый пункт — что изменилось + зачем. Объём пропорционален сессии
- Проекты — wikilinks внутри текста
- «Изменено:» — только ключевые файлы
- Решения и открытые вопросы — отдельными пунктами
- Пустая/тестовая сессия — не записывать

## Обновление записи (повторный вызов в той же сессии)

Одна техническая сессия может содержать несколько вызовов /session-log. Это нормально: пользователь фиксирует промежуточные итоги, потом продолжает работу.

При обновлении:
- Заголовок `###` → обнови время на текущее
- **Сессия:** → обнови метрики
- **Контекст:** → расширь, если появились новые домены
- Пункты работы → допиши новые, не трогай существующие
- «Изменено:» → объедини файлы
